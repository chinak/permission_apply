该仓库采用基于 React Context 的运行时状态管理模式，而非传统的静态配置文件（如 `.env` 或 `config.json`）。

### 1. 核心架构：Context 驱动的配置
- **全局状态即配置**：应用的“配置”（如用户风险等级、资金门槛、业务开关等）被定义为 React Context 中的状态。这种模式允许在运行时通过 UI 动态调整配置，非常适合演示环境或内部工具。
- **模块化隔离**：不同业务域拥有独立的 Context Provider：
  - `AppContext.tsx`：管理交易权限申请相关的通用配置（如 `riskLevel`, `fundLevel`, `investorType`）。
  - `WarehouseContext.tsx`：管理移仓业务特有的配置（如交易所选择、持仓方向、客户编码映射）。

### 2. 关键文件与逻辑
- **`src/app/store/AppContext.tsx`**：定义了权限申请模块的核心配置项，包括硬编码的默认账户 `85171680` 和初始风险等级 `C4`。
- **`src/app/store/WarehouseContext.tsx`**：定义了移仓业务的复杂表单状态，包含多个交易所（DCE, CZCE, SHFE）的编码映射及权限校验逻辑。
- **`src/app/components/ConfigPanel.tsx`**：提供了一个悬浮的“状态模拟面板”，允许开发者或测试人员在运行时手动修改 Context 中的配置值（如切换投资者类型、调整可用资金档位），从而实时观察业务逻辑的变化。

### 3. 部署与环境配置
- **构建配置**：使用 `vite.config.ts` 进行路径别名（`@` -> `./src`）和资源解析配置。
- **服务器配置**：通过 `deploy/nginx.conf` 和 `deploy/deploy.sh` 管理生产环境的 Nginx 行为，包括 SPA 路由回退、静态资源缓存策略及安全头设置。
- **工作区管理**：使用 `pnpm-workspace.yaml` 统一管理根目录及子应用（如 `permission_apply`）的依赖。

### 4. 开发规范与建议
- **避免硬编码**：目前 Context 中存在硬编码的测试数据（如账户号、客户名称）。在生产化过程中，应将这些初始值改为从 API 获取或通过环境变量注入。
- **配置与逻辑分离**：建议将复杂的业务校验规则（如适当性匹配逻辑）从组件中抽离，统一由 Context 或自定义 Hooks 管理，以保持 UI 层的简洁。
- **环境差异化**：当前缺乏明确的 `.env` 文件支持。若需区分开发、测试和生产环境，建议在 `vite.config.ts` 中引入 `loadEnv` 并定义 `VITE_` 前缀的环境变量。
